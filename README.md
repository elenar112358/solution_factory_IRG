### Почему lazy expiry не применяется к рекуррентным обязательствам  
*Бизнес-правило (lazy expiry). Перед формированием ответа сервис переводит все записи со статусом
active и next_payment_date < today в статус expired.
Рекуррентные обязательства (recurrence != null) под это правило не попадают: 
если пользователь не нажал «Оплачено», дата просто устарела, но подписка не
прекратилась. Для них статус остаётся active.*  

**Обоснование:**  
Рекуррентные обязательства всегда активны, даже устаревшие. 
Они не могут быть expired, по ним задолженность будет накапливаться до отмены.
Это необходимо учитывать при расчете текущей задолженности пользователя по обязательствам.
Задолженность погасится в базе только при внесении оплаты - и тогда будет сдвинута дата.
По факту по ней уже прошло списание денежных средств с карты (при их наличии) 
или произойдет, когда на карте что-то появится. 
В любом случае, эта ситуация должна вручную контролироваться пользователем.

### Как обрабатываются граничные случаи с датами при рекуррентности
При использовании ***dateutil.relativedelta*** корректно обрабатываются граничные даты:   
31 января + 1 месяц = 28 февраля  
Но есть особенность. Допустим, next_payment_date - это 31 января (последний день месяца), 
потом мы прибавляем месяц и получаем 28 февраля (последний день месяца).  
Если после этого мы еще раз прибавим месяц, то получим не 31 марта (последний день месяца),
а 28 марта. Далее оплаты будут ожидаться 28 числа каждого месяца.  
Если же мы прибавим сразу три месяца, то проскочим мимо 28-го и попадем на 30-е (апреля).  
Для того, чтобы поддерживать функционал "последний день месяца", 
необходим дополнительный параметр в модели данных.

### Какие компромиссы сделала и что бы сделала иначе при наличии большего времени
**Отклонения от ТЗ:**
1. **GET /obligations**  
По условию непонятно, условия на category и status как ***AND***, или ***OR***.  
Реализовано ***AND***.  
2. По условию эндпойнт POST /obligations/{id}/pay фиксирует факт оплаты и обновляет обязательство.  
Правило 1. Если статус не active - возвращает 422.  
Поскольку у EXPIRED обязательств тоже должна быть возможность оплаты, то
возвращает 422 только статус CANCELLED.  
Это согласуется с эндпойнтом GET /obligations/upcoming?days=N, где нет ограничений на EXPIRED.  
То есть в отчете по общей задолженности неоплаченные EXPIRED должны быть.  

**Расширение функционала**  
- Не реализован блок анализа платежей.  
Можно добавить отчет по платежам за период с детализацией по месяцам, кварталам, годам.  
Также детализация по категориям.  
- Можно предоставить пользователю отчет - план по предстоящим платежам с детализацией по периодам и категориям.  
- Дата оплаты сейчас - это момент занесения платежа в базу, а не фактический день оплаты.  
Это лишает возможности проводить дополнительный анализ о своевременности платежей 
(может быть актуально для нерекуррентных платежей).  
- Можно добавить функционал функционал "оплата в последний день месяца".  

### Примеры запросов curl

curl -X 'POST' \  
  'http://localhost:8000/api/v1/obligations/' \  
  -H 'accept: application/json' \  
  -H 'Content-Type: application/json' \  
  -d '{  
  "title": "string_9",  
  "amount": 10,  
  "currency": "RUB",  
  "category": "subscription",  
  "recurrence": "monthly",  
  "next_payment_date": "2026-07-13"  
}'  
  
curl -X 'GET' \  
  'http://localhost:8000/api/v1/obligations/?category=subscription&status=cancelled' \  
  -H 'accept: application/json'  
  
curl -X 'GET' \  
  'http://localhost:8000/api/v1/obligations/upcoming/?days=700' \  
  -H 'accept: application/json'  
  
curl -X 'POST' \  
  'http://localhost:8000/api/v1/obligations/7fb8e5ca-b568-46fd-930d-7ce5682304b2/pay' \  
  -H 'accept: application/json' \  
  -d ''  
  
curl -X 'PATCH' \  
  'http://localhost:8000/api/v1/obligations/318796a1-2ef3-4eb8-9fa1-db50867f4547/cancel' \  
  -H 'accept: application/json'  
  
curl -X 'DELETE' \  
  'http://localhost:8000/api/v1/obligations/318796a1-2ef3-4eb8-9fa1-db50867f4547' \  
  -H 'accept: */*'
